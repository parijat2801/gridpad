# Selection hierarchy under eager bands

Restore the four-level selection model — band, wireframe, shape, child — that the eager-bands refactor flattened to three. First click selects the outermost user-visible layer; repeat clicks drill in; ctrl+click bypasses to the deepest hit.

1. **Tree shape:** every claiming top-level frame is a `BAND` (`isBand: true`, full doc width, transparent, owns claim lines). The band's children are either a single shape (Figma-style: a freshly drawn rect lives directly under the band — `band → rect`) or a `WIREFRAME` container (`content: null`, tight bbox spanning all member shapes, holds two-or-more siblings — `band → wireframe → [rect, line, ...]`). Shapes carry their own children unchanged (text labels, nested rects). Solo shapes never get an implicit wireframe wrap; wireframes only exist where the scanner produced a multi-shape composite or the user explicitly groups.

2. **Restoring layer 2 in the loader.** `createEditorStateUnified` (editorState.ts:889-965) currently calls `wrapAsBand(absChildren, ...)` on `groupIntoContainers` output, lifting the container's children directly into the band and discarding the container — that's where layer 2 was lost. Fix: when the scanner-produced top-level frame has `content === null` (it IS a wireframe container with multiple children), keep it as a non-band `WIREFRAME` frame and put it inside the band as `wrapAsBand([wireframe], ...)`. When the scanner produces a solo shape (`content !== null`), wrap it directly with `wrapAsBand([shape], ...)` — no synthetic wireframe layer. Band geometry is unchanged: `gridRow`, `gridH`, `lineCount` still come from the wrapped child's bounds.

3. **Selection target rule (`resolveSelectionTarget(hit, currentSelectedId, frames, ctrlHeld)`):** with ctrl held, target = `hit.id`. Otherwise walk ancestors of `hit` from outermost-non-band inward, building `chain = [outermostNonBand, ..., hit]`. If `currentSelectedId` is one of `chain[0..n-2]`, target = the next deeper frame (`chain[indexOf(currentSelectedId) + 1]`) — drill-down. Otherwise target = `chain[0]` (outermost non-band). For `band → rect → text`, chain is `[rect, text]` — first click selects `rect`, second click drills to `text`. For `band → wireframe → rect → text`, chain is `[wireframe, rect, text]` — first click selects the wireframe, repeat clicks drill to the rect, then the text.

4. **Click handler integration.** `DemoV2.tsx:529-540` currently computes `hitContainer` by scanning `framesRef.current.find(f => !f.isBand && ...)` — only top-level non-bands, of which there are none under eager bands, so it always falls through to `hit.id`. Replace with `resolveSelectionTarget(hit, getSelectedId(state), framesRef.current, e.ctrlKey || e.metaKey)`. The existing double-click-to-edit branch (line 546) keeps its current trigger (`isDblClick && hit.content?.type === "text"`) — drilling all the way to the text via repeat clicks then double-clicking enters edit mode; ctrl+click on the text directly also reaches it in one step.

5. **Newly-drawn shapes (Figma-style intent at draw time).** `applyAddTopLevelFrame` (editorState.ts:1201) keeps its existing `findBandAtRow` shortcut: drawing into an existing band's rows still appends as a child of the band, not a new top-level. The change: if the cursor is over a wireframe container (`nestParent` resolved by `hitTestFrames` in DemoV2.tsx:482-498 lands on a frame whose parent is a wireframe), the new shape becomes a sibling inside that wireframe. If the cursor is over a solo rect, the new shape nests inside the rect (existing Figma-style behavior). If the cursor is over band whitespace, the new shape becomes a sibling of the existing wireframe/shape under the band — NOT a member. This delegates entirely to `nestParent` / `hitTestFrames`, no new logic; the only requirement is layer 2 must exist for the resolver to find it.

    **Band-grow for nested adds (per Gemini round 3):** `applyAddChildFrame` (editorState.ts:1285) currently doesn't auto-grow the containing band when nesting deep. With the 4-level model, drawing a shape into a wireframe whose bottom is at the band's bottom edge needs to push the band — same parity `applyAddTopLevelFrame` already has at editorState.ts:1229-1240. Fix: in `applyAddChildFrame`, walk to `findContainingBandDeep`; if the new child's absolute bottom (`getBandRelativeRow + child.gridH`) exceeds the band's `gridH`, emit a `resizeFrameEffect` on the band before the `addChildFrameEffect`.

6. **Serializer is depth-agnostic.** `serializeUnified.ts:79-115` already recurses children unconditionally (`for (const child of frame.children) renderFrameRow(child, localRow - child.gridRow, ...)`). The wireframe layer adds one more level between band and shape — recursion handles it. Add one round-trip test to confirm: load a multi-shape ASCII fixture, serialize, compare bytes.

7. **Drag/move/resize need ancestor-aware lookups AND band-relative coordinates (per Gemini reviews).** The current code assumes a shape's immediate parent IS the band — both for ancestor lookup AND for coordinate math. Both assumptions break with a wireframe in between.

    - **`findContainingBand` (editorState.ts:1402-1408)** — currently `f.isBand && f.children.some(c => c.id === frameId)`, only one level deep. Replace with a recursive `findContainingBandDeep(frames, frameId)` that walks all descendants (`hasDescendant`-style). Call sites at editorState.ts:613 (resize push physics) and 1341 (`applyReparentFrame`'s `sourceBand` lookup) now traverse intermediate wireframe layers.

    - **Coordinate relativity (NEW, per Gemini round 2):** `gridRow` and `gridCol` are *parent-relative* (frame.ts:558 — `wrapAsBand` rebases children with `gridRow: c.gridRow - minRow`). Under the 4-level model, a rect's `gridRow` is wireframe-relative, NOT band-relative. Any clamp or push-physics comparison against band bounds must first sum the chain of offsets. Use a single shared helper `findPath(frames, targetId): Frame[]` (top-down DFS returning the ancestor chain root → target). On top of `findPath`, derive `getBandRelativeRow(target, bandId, frames)` = sum of `gridRow` along the path from band to target. Same for col. (`findPath` is also reused by `resolveSelectionTarget` for its `chain` — one helper, two consumers.) Use these in:
        - **Push physics (editorState.ts:613-642):** replace `target.gridRow + Math.max(2, e.value.gridH)` with `getBandRelativeRow(target, parent, frames) + max(2, gridH)` before comparing to `parent.gridH`. Same for the column expression at line 616.
        - **Drag clamp (DemoV2.tsx:664-682):** replace `-child.gridRow` (which uses immediate-parent-relative coords) with `-getBandRelativeRow(child, containingBand, frames)`. Same for `parent.gridH - child.gridH - child.gridRow` etc — all four bounds.

    - **Drag clamp branch condition:** currently keys on `if (parent && parent.isBand)`. Replace with `const containingBand = findContainingBandDeep(framesRef.current, drag.frameId)`; if found, take the band-clamp branch using band-relative coords; otherwise the `else` branch (no claiming ancestor — pure relative motion) stands. The wireframe is part of the path but not the clamp boundary — only the band is.

    - **`findParentFrame`** stays as-is (returns immediate parent; correct for fluid-bbox recompute below).

8. **Wireframe bboxes auto-grow on child motion (NEW, per Gemini round 2).** Container frames have `clip: true` (frame.ts:573 for bands, frame.ts:119 for rects — wireframes inherit this) and `hitTestOne` (frame.ts:214-216) strictly rejects clicks outside the parent's `frame.x/y/w/h` before recursing. Without explicit recompute, dragging or resizing a child past its wireframe's bbox makes the child invisible AND un-hittable: the wireframe blocks both the paint pipeline and the click resolution. The plan's earlier "fluid by design" claim was wrong — there's no auto-recompute today.

    Fix: in the `framesField` reducer (editorState.ts:158+), after every `moveFrameEffect`, `resizeFrameEffect`, `addChildFrameEffect`, and `reparentFrameEffect` resolves, run a post-pass `recomputeWireframeBounds(frames)` that walks the tree and, for every `content === null && !isBand` frame, sets its `gridRow/gridCol/gridW/gridH` (and derived `x/y/w/h`) to the bounding union of its children's absolute extents. Call this once at the end of the reducer, AFTER `markDirtyById` and AFTER `mergeOverlappingBands`. Implementation note: the recompute is parent-relative-shift-aware — when a wireframe's `gridRow` changes (because the topmost child moved up), all its children's `gridRow` values need to rebase by the same delta to keep their absolute positions stable. This is the same logic `wrapAsBand` does at frame.ts:556-563. Extract it into a shared helper.

9. **Reparent must cascade through wireframes (per Gemini round 1).** The `reparentFrameEffect` reducer (editorState.ts:~348) currently prunes empty bands (`result = result.filter(f => !(f.isBand && f.children.length === 0))`) but not empty wireframes. Dragging the last shape out of a wireframe would leave an invisible 0-child container, and that container keeps the band alive forever. Fix: extend the prune step to remove `content === null && !isBand && children.length === 0` frames at any depth, then re-run the empty-band prune (a wireframe-emptying may now leave the band empty too — the existing band-prune handles that case once the wireframe is gone). Apply the same cascade to `applyReparentFrame`'s `sourceBandWillEmpty` calculation: a band "will empty" if removing the dragged shape leaves its only child wireframe empty (i.e., the wireframe was holding only this shape).

10. **`getFrameRects` test hook with absolute-coord accumulation (per Gemini round 2).** `DemoV2.tsx:808-833` currently unwraps bands into their immediate children, computing leaf position as `band.x + child.x`. With layer 2 present, the path is `band → wireframe → shape`, so the absolute position is `band.x + wireframe.x + shape.x` (triple sum, not double). Update the unwrap walk to recurse: while the current frame has `content === null` AND children, descend, accumulating `x/y` offsets at each level. Stop when reaching a frame with `content !== null` (the user-visible shape) and emit it. The harness's downstream pixel math relies on these absolute coords; off-by-one-layer here corrupts every drag.

11. **Ctrl+click affordance.** Detected via `e.ctrlKey || e.metaKey` in `onMouseDown` and threaded into `resolveSelectionTarget` as a single boolean parameter. No new state, no new effect — modifier resolves at click time. macOS users get cmd+click; everyone else ctrl+click.

12. **Test coverage — must include nested-mutation paths (per Gemini reviews).** Beyond the read-path tests (tree shape, `resolveSelectionTarget`, round-trip), add mutation tests that exercise the new wireframe layer specifically:

    - **Nested drag-against-band-edge:** load a multi-shape wireframe; drag one of its inner rects toward the band's top wall. Assert the rect clamps at the band wall (using band-relative coords, not wireframe-relative) and the band's residual-push fires (band gridRow shifts, doc rotates).
    - **Nested resize-grows-band:** resize an inner rect's height past the wireframe's current extent. Assert the wireframe's bbox grows AND the containing band's `lineCount` grows (push physics propagates through the wireframe layer with correct band-relative coordinate math).
    - **Nested extract-prunes-cascade:** drag the only inner rect out of a wireframe into empty doc space. Assert the wireframe is removed AND the parent band is removed AND the released claim lines are gone from the doc (no orphaned blank rows).
    - **Wireframe-bbox-recompute:** move a child rect within a wireframe (no band-edge interaction). Assert the wireframe's `gridRow/gridW/gridH` recompute to match the new union of children. Then move the rect back; wireframe shrinks back. Hit-test the moved rect at its new location — must succeed (clip-and-hit-test work after recompute).
    - **Band-relative coord helper:** unit test `getBandRelativeRow(shape, band, frames)` for all four tree shapes — solo, multi-shape, deeply-nested-rect-in-rect, deeply-nested-rect-in-wireframe-in-rect. Assert sums match expected values; assert returns sentinel (or throws) when `band` is not an ancestor.
    - **`findContainingBandDeep` unit:** three cases (immediate child, grandchild via wireframe, not present).

13. **Migration of the 25 failing harness tests.** The 5 "session-introduced" failures all click box centers — under the new model first click selects the wireframe (or solo rect), drag moves the wireframe (or rect), `Bottom` text stays inside. No harness changes required. The 19 baseline-failing tests are a separate triage; some may turn green incidentally (anything that depends on first-click selecting the rect not the text), some won't.
